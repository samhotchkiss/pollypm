# T064: Alert Created and Stored in SQLite

**Spec:** v1/10-heartbeat-and-supervision
**Area:** Alerting
**Priority:** P1
**Duration:** 10 minutes

## Objective
Verify that alerts generated by the heartbeat system are stored in the SQLite database with all required fields and are queryable.

## Prerequisites
- `pm up` has been run
- At least one alert has been generated (or trigger one by killing a session)

## Steps
1. Trigger an alert if none exist: kill a worker process (`kill -9 <PID>`) and wait for the heartbeat to detect it.
2. Run `pm alert list` to see the generated alerts via CLI.
3. Locate the SQLite database: check `.pollypm/state.db` or `pm config show` for the database path.
4. Open the database: `sqlite3 <path-to-state.db>`.
5. Run `.tables` and identify the alerts table.
6. Query the alerts: `SELECT * FROM alerts ORDER BY timestamp DESC LIMIT 10;`.
7. Verify each alert record contains:
   - Alert ID (unique)
   - Alert type (e.g., "session_exited", "session_stuck", "session_looping")
   - Session ID that triggered the alert
   - Timestamp
   - Severity level
   - Description/message
   - Status (e.g., "open", "acknowledged", "resolved")
8. Verify the alert IDs are unique (no duplicates).
9. Generate a second alert and query again — verify it appears with a new ID and later timestamp.
10. Test querying by type: `SELECT * FROM alerts WHERE type = 'session_exited';` — verify it returns only the relevant alerts.

## Expected Results
- Alerts are stored in the SQLite database
- Each alert has all required fields (ID, type, session, timestamp, severity, message, status)
- Alert IDs are unique
- Alerts are queryable by type, session, timestamp, and status
- `pm alert list` output matches the database contents
- New alerts appear immediately in the database
